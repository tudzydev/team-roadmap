# คู่มือการใช้งาน Git ในสถานการณ์จริง: Self, Team, & Enterprise

คู่มือนี้แบ่งปันแนวปฏิบัติในการใช้ Git ที่ใช้ในการทำงานจริง โดยแบ่งตามระดับและขนาดของโปรเจกต์ เพื่อเป็นแนวทางที่เหมาะสมและมีประสิทธิภาพสูงสุด

---

## 👤 1. Self Project (โปรเจกต์ส่วนตัว / ทำงานคนเดียว)
**แนวคิดหลัก:** ความเรียบง่าย ความเร็ว และการเก็บันทึกประวัติเพื่อย้อนกลับได้เมื่อเกิดปัญหา (Version Tracking & Backup)

### 📂 เวิร์กโฟลว์ (Workflow)
ใช้ **Single Branch (Main Branch)** หรือ **GitHub Flow (แบบย่อ)**:
* เขียนโค้ดและ Commit ตรงเข้าสู่กิ่ง `main` ได้โดยตรง
* แยก Branch ย่อยชั่วคราว เฉพาะตอนทดลองฟีเจอร์ใหม่ที่มีความเสี่ยงสูง (เมื่อเสร็จก็ Merge กลับเข้า `main`)

### 🛠️ แนวปฏิบัติที่ดีที่สุด (Best Practices):
1. **ตั้งค่า `.gitignore` ตั้งแต่วันแรก:**
   * ห้ามนำโฟลเดอร์ที่ไม่จำเป็นขึ้น Git เช่น `node_modules/`, `.env` (ที่เก็บคีย์ลับ), หรือโฟลเดอร์ build เช่น `dist/`, `build/`
2. **เขียน Commit Message ให้เข้าใจง่าย (Conventional Commits):**
   * แม้ทำคนเดียว ก็ควรเขียนคอมมิตตามรูปแบบมาตรฐานเพื่อให้สืบย้อนประวัติได้ง่าย เช่น:
     * `feat: add Google Login support`
     * `fix: resolve crash on null memory value`
3. **ใช้ Semantic Tagging สำหรับออกเวอร์ชัน:**
   * เมื่อโปรเจกต์ใช้งานได้ดี ให้ปักป้ายเวอร์ชัน (Tag) ไว้ เพื่อให้ย้อนกลับมาดาวน์โหลดโค้ดชุดนี้ได้เสมอ:
     ```bash
     git tag -a v1.0.0 -m "Release version 1.0.0"
     git push origin v1.0.0
     ```

## 👥 2. Team Project (โปรเจกต์ร่วมกันในทีมขนาดกลาง)
**แนวคิดหลัก:** ป้องกันไม่ให้โค้ดทับกัน การควบคุมคุณภาพผ่านระบบรีวิว และการรักษาเสถียรภาพของสายหลัก (Quality Control & Branch Protection)

### 📂 เวิร์กโฟลว์ (Workflow)
ใช้ **GitHub Flow (Feature Branching Workflow)**:
1. ดึงโค้ดล่าสุดจาก `main` ➔ สร้าง Branch ย่อยเพื่อพัฒนาฟีเจอร์
2. พัฒนาเสร็จ ➔ Push ขึ้น GitHub ➔ เปิด **Pull Request (PR)**
3. สมาชิกในทีมทำการ **Code Review** และรันเทส ➔ เมื่ออนุมัติจึงทำการ **Merge** เข้า `main`

### 🛠️ แนวปฏิบัติที่ดีที่สุด (Best Practices):
1. **ตั้งชื่อกิ่งตามมาตรฐาน (Branch Naming Convention):**
   * ระบุประเภทงานให้ชัดเจน เช่น:
     * `feat/user-profile` (เพิ่มฟีเจอร์ใหม่)
     * `fix/password-validation` (แก้บั๊ก)
     * `docs/readme-setup` (ปรับปรุงคู่มือ)
2. **เปิดระบบป้องกันสายหลัก (Branch Protection Rules):**
   * ตั้งค่าบน GitHub ห้ามสมาชิกทุกคน (รวมถึง Admin) ทำการ Push ตรงไปยัง `main`
   * บังคับว่าการรวมโค้ดต้องผ่านการรีวิวและอนุมัติจากสมาชิกในทีมอย่างน้อย 1 คน (`Require status checks / Approvals`)
3. **การทำ Squash and Merge:**
   * ตอน Merge PR ให้ใช้โหมด **Squash and Merge** เพื่อรวมคอมมิตย่อยๆ หลายตัวของการพัฒนาแอป (เช่น *fix bug, test 1, test 2*) ให้เหลือเพียง Commit เดียวที่มีสาระสำคัญเมื่อเข้าสู่กิ่งหลัก เพื่อให้ประวัติ `git log` สะอาดอ่านง่าย
4. **สื่อสารก่อนทำการ Rebase/Merge:**
   * สมาชิกควรทำการดึงโค้ดล่าสุดจาก main มา rebase เข้ากิ่งตัวเองบ่อยๆ เพื่อแก้ไข Conflict บนกิ่งของตัวเองก่อนจะส่ง PR เข้ากิ่งหลัก
     ```bash
     git switch feat/my-feature
     git fetch origin
     git rebase origin/main
     ```

## 🏢 3. Enterprise Project (โปรเจกต์ระดับองค์กรขนาดใหญ่)
**แนวคิดหลัก:** ความปลอดภัยสูง การคุมสิทธิ์ที่รัดกุม การรองรับ Repository ขนาดใหญ่ และระบบจัดส่งซอฟต์แวร์อัตโนมัติ (High Compliance, Scale, & Automation)

### 📂 เวิร์กโฟลว์ (Workflow)
ใช้ **Trunk-Based Development** หรือ **Git Flow (แบบมี Release Branch)**:
* **Trunk-Based:** นักพัฒนาทุกคนนำงานชิ้นเล็กๆ เข้าสู่ `main` ทุกวัน โดยฟีเจอร์ที่ยังไม่พร้อมใช้งานจะถูกซ่อนอยู่หลัง **Feature Flags** (ช่วยให้รวมโค้ดเร็วขึ้นและลด Merge Conflicts ระยะยาว)
* **Release Branching:** แยกกิ่ง release เช่น `release/v2.1` เพื่อทดสอบแก้บั๊กขั้นสุดท้ายก่อนออกใช้งานจริง ขณะที่นักพัฒนาคนอื่นก็พัฒนาเวอร์ชันถัดไปได้ต่อโดยไม่ถูกบล็อก

### 🛠️ แนวปฏิบัติที่ดีที่สุด (Best Practices):
1. **ระบบเจ้าของโค้ด (CODEOWNERS):**
   * สร้างไฟล์ `.github/CODEOWNERS` เพื่อระบุกลุ่มหรือผู้ดูแลที่ต้องรีวิวโค้ดในแต่ละโฟลเดอร์โดยเฉพาะ เช่น:
     ```text
     # ทีม Infra ต้องอนุมัติทุกการแก้ไฟล์ในห้อง terraform
     /terraform/ @enterprise-infra-team
     # ทีม Security ต้องอนุมัติการแก้ไขระบบ Auth
     /src/auth/ @enterprise-security-team
     ```
2. **การเซ็นชื่อดิจิทัลยืนยันตัวตน (Signed Commits):**
   * บังคับให้นักพัฒนาทุกคนในองค์กรเชื่อมต่อ SSH หรือ GPG Key เพื่อเซ็นรับรอง (Sign) ทุก Commit ป้องกันการสวมรอยส่งโค้ดแปลกปลอมเข้ามาในโครงการ
3. **ระบบตรวจสอบ CI/CD แบบ 100% Pass:**
   * ห้ามรวมโค้ดหากระบบเช็กอัตโนมัติ (Security Scan, SonarQube Quality Gate, Integration Tests) ยังรันไม่ผ่าน
4. **การคุมไฟล์ขนาดใหญ่ด้วย Git LFS (Large File Storage):**
   * องค์กรที่มีการใช้ Assets ขนาดใหญ่ (รูปดิบ, ไฟล์วิดีโอ, โมเดล AI) ต้องใช้ Git LFS เพื่อป้องกันไม่ให้ขนาดโฟลเดอร์ `.git` บวมจนเครื่องทำงานช้าลง
5. **วิเคราะห์ Monorepo vs. Polyrepo:**
   * วางแผนร่วมกันว่าควรเก็บโค้ดระบบทั้งหมดไว้รวมกัน (Monorepo) หรือแยกตามไมโครเซอร์วิส (Polyrepo) โดยมีเครื่องมือช่วยจัดการ เช่น Nx, Lerna หรือ Turborepo
